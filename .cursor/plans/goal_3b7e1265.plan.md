### Goal

Use `next-intl` for en/ar/so translations and RTL support across your existing `app/` tree (no `src/`), integrating with the current language picker in `app/(home)/components/drawer-theme.tsx`.

### 1. Core i18n wiring for current structure

- **Locale config file**: Create a small config module at project root (for example `[project]/i18n.ts`) that exports `locales = ['en','ar','so']`, `Locale`, and `defaultLocale = 'en'`.
- **Shared navigation helper**: Add `[project]/navigation.ts` using `createSharedPathnamesNavigation` from `next-intl/navigation`, importing `locales` from `./i18n`, and exporting `Link`, `useRouter`, `usePathname`, and `redirect`.
- **Restructure layouts for locales**:
  - Move the current `app/layout.tsx` content to `app/[locale]/layout.tsx`, adjusting imports.
  - Keep your segment layouts (`app/(home)/layout.tsx`, `app/(auth)/layout.tsx`) nested under `app/[locale]/` if needed, or ensure they work correctly under the new locale segment.
- `**NextIntlClientProvider` + RTL** in `app/[locale]/layout.tsx`:
  - Implement `generateStaticParams` returning `{ locale }` for all `locales`.
  - Load messages dynamically from `./messages/{locale}.json` (for example using `import('@/messages/en.json')` style but adjusted to your final path).
  - Validate `params.locale` against `locales`, call `notFound()` when invalid, and set `<html lang={locale} dir={locale === 'ar' ? 'rtl' : 'ltr'}>`.
  - Wrap `children` with `NextIntlClientProvider`.

### 2. Message files and key design

- **Message file location**: Add a `messages/` folder at the project root: `messages/en.json`, `messages/ar.json`, `messages/so.json`.
- **Scan `app/` for visible text**: Go through files under `app/`, especially:
  - `app/(home)/page.tsx`, `app/(home)/layout.tsx`, `app/(home)/components/*` (sidebar/menu, table, dialogs, logo, etc.).
  - `app/(auth)/layout.tsx`, `app/(auth)/login/page.tsx`, `app/(auth)/components/*` (login, verify forms, auth card).
- **Define key groups**: Design consistent keys like `nav.dashboard`, `nav.hrm`, `nav.finance`, `home.welcome`, `employees.title`, `auth.loginTitle`, `auth.emailLabel`, etc., based on real text you find.
- **Populate translations**: Fill en/ar/so values for every key; Arabic gets RTL text; Somali stays LTR.

### 3. Connect `drawer-theme.tsx` language picker

- **Understand current behavior**: Inspect `app/(home)/components/drawer-theme.tsx` to see how it stores the selected language (state, context, localStorage).
- **Switch to locale-aware navigation**:
  - Import `useRouter` and `usePathname` from your new `navigation.ts`.
  - On language change, compute the same path but under the new `[locale]` segment (e.g. replace `/en/...` with `/ar/...`) and call `router.replace(...)` or `router.push(...)`.
  - Ensure that when there is no locale prefix (current state), you route to `/${defaultLocale}` consistently.
- **Make picker text translatable**: Any labels inside the drawer related to language/theme should use `useTranslations` and message keys (for example `settings.language`, `settings.theme.system`).

### 4. Replace hard-coded strings in current components

- **Home and shared components**:
  - In `app/(home)/page.tsx`, replace headings, descriptions, and section titles with `t(...)` calls from `useTranslations`.
  - In `app/(home)/components/menu-list.ts`, `drawer-theme.tsx`, `logo.tsx`, `vertical.tsx`, `horizantal.tsx`, `table.tsx`, and `confirmation-dialog.tsx`, replace menu item labels, button texts, dialog titles, and any visible strings with message keys.
- **Auth flows**:
  - In `app/(auth)/login/page.tsx`, `auth-card.tsx`, `login-form.tsx`, and `verify-form.tsx`, replace all user-facing labels, placeholders, and button texts with `useTranslations`.
- **Keep changes minimal**: Where possible, only wrap the string with `t('...')` and add the key to JSON, avoiding any refactors beyond what is needed for i18n.

### 5. RTL-safe styling in existing layouts

- **Use logical Tailwind utilities**:
  - In layout components (including `globals.css` and shared layout files), update directional styles: `ml-`/`mr-` ŌåÆ `ms-`/`me-`, `pl-`/`pr-` ŌåÆ `ps-`/`pe-`, `text-left`/`text-right` ŌåÆ `text-start`/`text-end` where alignment should flip for Arabic.
- **Sidebar and menus**:
  - In navigation/menus (likely `menu-list.tsx` and related components), use `ms-auto`, `ms-4`, `border-s`, `start-0`/`end-0` instead of hard left/right so they flip automatically with `dir`.
  - For chevrons/arrows, add `rtl:rotate-180` where they indicate direction.
- **Global RTL font**:
  - In `app/globals.css`, add an optional `[dir="rtl"]` rule to apply an Arabic-friendly font stack when `dir='rtl'`.

### 6. Validate behavior per locale

- **Manual checks for `/en`, `/ar`, `/so`**:
  - Ensure routes like `/en/(home)` and `/en/(auth)/login` render correctly with English text and LTR.
  - Switch to Arabic ŌåÆ check all main screens for Arabic strings, RTL direction, flipped paddings/margins, and mirrored icons.
  - Check Somali routes for correct Somali strings with LTR layout.
- **Fix missing/mismatched keys**: Whenever a component shows a fallback or missing translation, add/adjust the key in all three message files and keep the key set clean and consistent.
